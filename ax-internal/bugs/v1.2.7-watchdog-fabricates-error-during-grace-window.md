Classification: confirmed

# v1.2.7 still shows "The request was accepted, but no assistant response or error was produced"

## Summary

The accepted-prompt watchdog fabricates a synthetic error when an SSE `session.idle` /
`session.status: idle` event transiently clobbers the optimistic busy status to idle
during the 30s prompt-accepted grace window. The grace window
(`wasPromptRecentlyAccepted` / `PROMPT_ACCEPTED_BUSY_GRACE_MS = 30s`) only guards the
**status-poll / reconnect** path via `resolveResyncedSessionStatus`. It does **not**
guard the SSE event-reducer path, which writes status events directly to the store.

## Evidence

1. Error string source: `packages/ui/src/sync/session-actions.ts:284` — fabricated by the
   watchdog in `scheduleAcceptedPromptWatchdog` (lines 260-300) when, after 12s, there is
   no assistant response AND `isSessionWorking` returns false (status not busy/retry).

2. Grace window only guards the poll path:
   - `wasPromptRecentlyAccepted` is consumed only in `sync-context.tsx:525`
     (`buildRelevantSessionStatuses` → `resolveResyncedSessionStatus`), which runs on the
     periodic status poll and reconnect resync.
   - `resolveResyncedSessionStatus` (`reconnect-recovery.ts:37-54`) preserves the existing
     non-idle status when `promptRecentlyAccepted && !hasCompletedAssistantReply`.

3. The SSE event-reducer does NOT consult the grace window:
   - `event-reducer.ts:426-433` (`session.status`) writes `status` directly to
     `session_status[sessionID]`.
   - `event-reducer.ts:435-443` (`session.idle`) writes `{ type: "idle" }` directly.
   - `event-reducer.ts:445-453` (`session.error`) writes `{ type: "idle" }` directly.
   None of these check `wasPromptRecentlyAccepted`.

4. The watchdog (12s) fires well inside the grace window (30s), so
   `wasPromptRecentlyAccepted` is ALWAYS true when the watchdog runs in normal operation.
   Yet the watchdog's fabrication branch (lines 256-300) never checks the grace window —
   it only checks `isSessionWorking(store, sessionId)`.

5. Prior fixes (v1.2.6, v1.2.7) covered **cross-turn** clobbering (stale watchdog from a
   previous prompt) and the synchronous markPromptAccepted race, but neither covered a
   **same-turn** SSE idle/status event arriving during the grace window.

## Root cause / trigger sequence

1. User sends a prompt. `optimisticSend` sets busy + `markPromptAccepted` + schedules the
   12s watchdog.
2. During the next 12s, the AX Code server emits a `session.idle` (or `session.status:
   idle`) SSE event — e.g. the session was momentarily idle between the prompt_async
   return and the turn spinning up, or a stale/reordered status snapshot.
3. `event-reducer` writes idle directly to `session_status[sessionId]`, clobbering busy.
   (The poll path would have been protected, but the reducer path is not.)
4. At t=12s the watchdog fires: no assistant response, status=idle.
5. `recoverAcceptedPromptFromServer` refetches; if the turn's events are still in flight /
   not yet persisted (or the directory routing misses), it finds nothing.
6. Watchdog fabricates the synthetic error and forces idle. User sees the error.

## Suggested Fix

In `scheduleAcceptedPromptWatchdog`, the **fabrication branch** (the final `else` that
creates the synthetic error message) must defer to the grace window: if
`wasPromptRecentlyAccepted(sessionId)` is still true when the watchdog fires, do NOT
fabricate — instead re-arm the watchdog to fire again after the grace window expires
(`PROMPT_ACCEPTED_BUSY_GRACE_MS - elapsed`), so a genuinely dead turn is still caught but
a transient grace-window idle clobber does not produce a false error.

The initial-match and recovery idle-forcing branches (lines 209-226, 245-252) already have
the cross-turn `hasNewerUnansweredUserMessage` guard; the same grace-window deferral should
apply there too for consistency, but the fabrication branch is the one that produces the
user-visible error string.
