# v1.2.8 Watchdog Error Investigation Report

## Problem Statement
User reports that v1.2.8 still shows "The request was accepted, but no assistant response or error was produced" error on the second prompt, while v1.2.3 does not have this problem.

## Key Differences Between v1.2.3 and v1.2.8

### 1. Grace Window Mechanism (Added in v1.2.4+)
**v1.2.3**: No grace window mechanism
**v1.2.8**: Has `markPromptAccepted()`, `wasPromptRecentlyAccepted()`, and `acceptedPromptAt` map

**Purpose**: Prevent status poll/reconnect from clobbering optimistic busy status to idle during the 30s window after prompt acceptance.

**Potential Issue**: The grace window only guards the status-poll path (`resolveResyncedSessionStatus`), NOT the event-reducer's direct status writes from SSE events.

### 2. Server Recovery in Watchdog (Added in 0648010f)
**v1.2.3**: Watchdog fabricates error immediately if no local response and status is idle
**v1.2.8**: Watchdog calls `recoverAcceptedPromptFromServer()` before fabricating

**Purpose**: Recover real assistant response from server if SSE events were dropped.

**Potential Issue**: Server recovery is async and takes time. During this time, the status might change.

### 3. Stale Watchdog Cancellation (Added in 9d672c94)
**v1.2.3**: No cancellation of previous watchdog
**v1.2.8**: Cancels previous watchdog timer when new prompt is sent

**Purpose**: Prevent previous prompt's watchdog from interfering with current prompt.

**Potential Issue**: `clearTimeout()` only prevents the timer from firing. If the timer has already fired and the callback is running, it won't stop the callback.

### 4. Newer Unanswered User Message Guard (Added in 9d672c94)
**v1.2.3**: No guard against cross-turn clobbering
**v1.2.8**: `hasNewerUnansweredUserMessage()` prevents forcing idle when newer prompt is in progress

**Purpose**: Prevent stale watchdog from forcing idle during a newer prompt's turn.

**Potential Issue**: Only guards the "force idle" branches, not the "fabricate error" branch.

### 5. Grace Window Re-arm (Added in 0a6bc05b - v1.2.8)
**v1.2.3**: No re-arm
**v1.2.8**: If watchdog fires within grace window, re-arms to fire again after grace expires

**Purpose**: Prevent false errors from transient SSE idle events during grace window.

**Potential Issue**: Re-arm delay calculation might be incorrect, or `wasPromptRecentlyAccepted()` might return false when it should return true.

## Hypothesis

The most likely root cause is that **an SSE `session.idle` or `session.status:idle` event is clobbering the optimistic busy status to idle during the grace window**, and the watchdog is fabricating an error because:

1. The grace window only guards the status-poll path, not the event-reducer path
2. The watchdog checks `isSessionWorking()` which returns false if status is idle
3. The watchdog checks `wasPromptRecentlyAccepted()` and re-arms if true
4. But if the re-armed watchdog fires after the grace window expires (~30s), and there's still no assistant response, it fabricates an error

## Why v1.2.3 Works

In v1.2.3, there was no grace window mechanism, so the watchdog would fire at 12s and fabricate an error if there was no response. But the user says v1.2.3 works, which means the assistant response WAS arriving within 12 seconds.

So the assistant response should also be arriving within 12 seconds in v1.2.8. But the watchdog is still fabricating an error, which suggests:

1. The assistant response is not being found by the watchdog (wrong parentID, not in store, etc.)
2. The status is being set to idle incorrectly, causing the watchdog to proceed to fabrication
3. The server recovery is not finding the response
4. The grace window re-arm is not working correctly

## Recommended Investigation Steps

1. **Add detailed logging** to the watchdog to see:
   - When it fires
   - What status it sees
   - Whether it finds an assistant response
   - Whether `wasPromptRecentlyAccepted()` returns true
   - Whether it re-arms or fabricates

2. **Check if SSE events are clobbering status**:
   - Add logging to event-reducer when it processes `session.idle` or `session.status` events
   - Check if these events arrive during the grace window

3. **Check if assistant response is arriving**:
   - Add logging when assistant messages are added to the store
   - Check if the parentID matches the user message ID

4. **Check if server recovery is working**:
   - Add logging to `recoverAcceptedPromptFromServer()` to see what it fetches
   - Check if the server has the assistant response

5. **Test with increased timeout**:
   - Temporarily increase `ACCEPTED_PROMPT_NO_OUTPUT_MS` from 12s to 60s
   - See if the error still occurs

## Conclusion

The root cause is likely that the grace window mechanism is incomplete. It guards the status-poll path but not the event-reducer path, allowing SSE events to clobber the optimistic busy status during the grace window. The watchdog then fabricates an error because it sees idle status and no assistant response.

The fix should either:
1. Guard the event-reducer path with the grace window (prevent SSE idle events from clobbering busy during grace window)
2. Make the watchdog more lenient (don't fabricate error if prompt was recently accepted, even if status is idle)
3. Remove the watchdog entirely and rely on other mechanisms to detect stuck prompts
